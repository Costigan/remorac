# Descriptor Export Lowering Gap

## Status

The descriptor export path (`_lower_function_descriptor_module` → `_lower_descriptor_internal_function`) cannot lower gradient functions
generated by `generate_gradient_function_source`. This blocks the compiled CPU training path
(`CPUFunctionExecutor`).

All other compilation paths work: standalone expressions via `evaluate_source_compiled`, simple
named functions via `compile_function_source`, and interpreter execution via `_lambda_callable`.

## Symptoms

`compile_function_source(gradient_source, ...)` returns empty MLIR (caught `RemoraLoweringError`).

The error chain starts with `unbound HIR variable b1` in `remora/lowering/scalar.py:209`.

## Root Cause

The gradient function body mixes **scalar** and **array** operations in a single expression tree.
For example, a CNN gradient contains:

```
(reshape (fold + (with-shape 0.0 [9]) (map * (transpose ...) ...)) [3 3])
```

The fold's init `(with-shape 0.0 [9])` is an array operation. The fold's accumulator `+` and
the `*` inside the map are scalar operations. The body also references scalar parameters
(`b1: Float`, `b3: Float`, `y: Float`) from the function signature.

`_lower_descriptor_internal_function` sets up two separate environments:

```python
scalar_env: dict[str, _Operand]   # maps param name → SSA value for scalar params
tensor_env: dict[str, _TensorValue]  # maps param name → SSA value for array params
```

The fix applied in this session added scalar params to `tensor_env` as well (as 0-d tensors),
so `HIRVar(b1)` can be resolved by `_lower_tensor_input`. However, when `_lower_tensor_input`
dispatches to scalar lowering (`_lower_scalar_module`), that path does **not** receive
`scalar_env`. Scalar operations that reference `b1` fail with `unbound HIR variable`.

## What Needs to Change

Thread `scalar_env` alongside `tensor_env` through the lowering chain. Functions that
currently accept `(node, functions, prefix, tensor_env=None)` need to also accept
`scalar_env=None`. The functions are in:

| File | Functions |
|------|-----------|
| `remora/lowering/tensor_ops.py` | `_lower_fold_input`, `_lower_fold_result`, `_lower_scalar_fold_result`, `_lower_array_fold_result`, `_lower_map_cell_result`, `_lower_binary_map_fold_input`, `_lower_fold_callable_body`, `_lower_map_callable_body`, and helpers |
| `remora/lowering/view_ops.py` | `_lower_view_input`, `_lower_view_result`, `_lower_transpose_result`, `_lower_reshape_result`, `_lower_reverse_result`, `_lower_take_result`, `_lower_drop_result` |
| `remora/lowering/module.py` | `_lower_descriptor_scalar_result_body`, `_lower_scalar_value_for_fold_init`, `_lower_descriptor_internal_function` (callers) |
| `remora/lowering/scalar.py` | `_lower_scalar_module` (the error site — needs `scalar_env` to resolve `HIRVar`) |

Each function passes `scalar_env` (or `None`) to its callees. The `_lower_descriptor_internal_function`
already creates both environments; it just needs to pass them through.

## Pre-requisites Already Done

These fixes were applied and tested (all 962 non-training tests pass):

1. **HIRLet inlining** — `_inline_lets` in `module.py` now handles all HIR types (HIRWithShape, 
   HIRIm2col, HIRCol2im, HIRScatterAdd, HIRTranspose, HIRReshape, HIRRavel, HIRReverse, 
   HIRTake, HIRDrop, HIRRotate, HIRSubarray, HIRAppend). This prevents let-bound HIRVar
   references from leaking into lowering.

2. **Scalar params in tensor_env** — `_lower_descriptor_internal_function` adds scalar-typed
   function parameters to `tensor_env` as 0-d tensors, so `_lower_tensor_input` can resolve them.

3. **HIRIf in `_lower_tensor_input`** — Array-typed `HIRIf` nodes (generated by `select` in
   the gradient source) are lowered via `linalg.generic` with `arith.select`.

4. **Fold input fallback** — `_lower_fold_input` falls back to `_lower_tensor_input` for
   unsupported expression types instead of raising `RemoraLoweringError`.

5. **Index range fixes** — All `range(max_idx)` bounds in `_lower_im2col_tensor_input` and
   `_lower_col2im_tensor_input` use `range(max_idx + 1)` with correct `max_idx` computation
   covering all patch and pixel indices.

## Acceptance Criteria

```python
result = compile_function_source(gradient_source, function_name, param_types,
                                  include_prelude=False, syntax='lisp', verify=False)
assert result.mlir_text != ''  # MLIR generated successfully
```

Then:
```python
executor = CPUFunctionExecutor.compile_source(gradient_source, function_name, param_types,
                                                include_prelude=False, syntax='lisp')
executor = CPUFunctionExecutor(artifact)
result = executor.execute(k, b1, w2, b2, w3, b3, mask, x, y)
assert result is not None  # compiled gradient executes correctly
```
