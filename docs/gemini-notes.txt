> What is session-level arena management?
▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄
✦ Session-level arena management is the strategy of using a single, persistent memory pool for all computations within a lifetime of a Remora session (such as a REPL instance or a
long-running application loop), rather than allocating fresh memory for every individual command.

Currently, we have the "Arena Allocator foundation," which means we can pass an Arena to an executor. Session-level management takes this further in three key ways:

1. Persistence of Data (Keeping it "Warm")
In a REPL or interactive session, you often do this:

 1 let x = iota 1000000
 2 let y = map (* 2) x
Without session-level management, x might be copied back to the CPU or its device memory freed after the first line. With a session arena:
 * x is allocated once in the Session Arena.
 * The second line (map (* 2) x) finds x already sitting in the arena.
 * Zero host-to-device copies are needed for x because it never left the GPU.

2. Eliminating Allocation Latency
Allocating memory on a GPU (cudaMalloc) is a relatively slow, synchronous operation. In a high-performance loop, doing this hundreds of times per second kills performance.
 * Session-level: We cudaMalloc one large block (e.g., 2GB) when the Remora session starts.
 * All subsequent operations just bump a pointer inside that 2GB block.
 * Allocation becomes effectively instantaneous.

3. Predictable Memory Capping
By managing memory at the session level, the compiler can act more like a traditional runtime.
 * You can launch Remora with a flag like --memory-limit 4G.
 * The session arena ensures the compiler never uses more than that limit, providing safety for shared environments or constrained hardware.

4. Automated "Compaction" or Resetting
Since Remora uses a functional paradigm (immutable arrays), memory usage can grow quickly. A session-level manager can track which variable bindings are still active. When the arena
fills up, it can:
 * Identify which buffers are "garbage" (no longer bound to a name).
 * Perform a compaction (move live arrays to the start of the arena).
 * Reset the pointer, allowing the session to continue indefinitely without a malloc "leak."

What this looks like for us next:
To implement this, we would likely introduce a Session object in remora/runtime.py that ties together:
 1. A Persistent Arena (for CPU or GPU).
 2. A Symbol Table (mapping variable names to pointers inside that arena).
 3. An Automatic Reset/Cleanup trigger used by the REPL or CLI.

This turns the compiler from a "one-shot" tool into a persistent execution environment. Would you like to begin implementing this Session architecture?
